# Pull Request Size

Big PRs are slow. They block reviewers, hide bugs, and stall merges. Every PR ships small or splits.

## Hard rule

- **Target:** ≤ 16 hand-written file changes per PR.
- **Cap:** ≤ 20 hand-written file changes per PR.
- **Over the cap →** split into `Part 1`, `Part 2`, … PRs. Each part ships independently and passes CI on its own.

## What counts as a file change

**Counts:**
- Created files (any hand-written source, test, template, doc).
- Modified files (any line change).
- Deleted files.
- Renames (count as 1 each).

**Does not count:**
- Prisma migration SQL files (auto-generated by `prisma migrate`).
- `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`.
- Snapshot files written by Jest/Vitest (`__snapshots__/*.snap`).
- Generated client output (e.g. `node_modules/.prisma/`, `dist/`, build artifacts — these should never be in a PR anyway).
- The `package.json` line that bumps a dep version added in this PR.

If the auto-generated set materially changes the diff (e.g. a 2000-line migration), call it out in the PR description. Otherwise it's invisible to the count.

## Splitting strategy

Decide the split at **planning time**, not mid-execution.

- The plan author bounds Part 1 to fit the cap.
- Subsequent parts are listed in the same plan file as `## Part 2`, `## Part 3`, … sections, each with its own task list.
- Each part lists its file budget at the top: `**Part 1:** 15 files. **Part 2 follow-up:** 12 files.`
- Part 1 must produce **working, testable software on its own** — not a half-built feature that 404s without Part 2. Choose the seam carefully (e.g., domain + first provider in Part 1; second provider in Part 2; UI in Part 3).
- Part 2's branch is `feat/<feature>-part-2` (off the same base — usually `main`, after Part 1 merges).

## Naming and tracking

- **Spec title:** `Feature Name — Design (Part 1: <slice>)`
- **Plan title:** `Feature Name — Implementation Plan (Part 1)`
- **Branch:** `feat/<feature>` for Part 1; `feat/<feature>-part-2` for follow-ups.
- **PR title:** `feat(area): <feature> (part 1/N)` — always include the `(part X/N)` suffix when splitting.
- **PR description:** link to subsequent parts (`Part 2: #<pr>`) and the original spec.
- **Follow-up state:** maintain a tracker file at `docs/superpowers/follow-up-prs/<feature>.md` (see template in that directory). This is the bookmark you read when picking the work back up days later.

## Atomic-refactor escape hatch

Some changes can't be split without a feature-flag dance that adds more code than the rename it's hiding (e.g., renaming a symbol used in 40 files; deleting a deprecated module; bumping TypeScript major and fixing all the type errors).

**Allowed if:**
- The change is genuinely atomic — splitting requires temporary scaffolding that gets removed in a follow-up.
- The PR description includes a section `## Why this PR is over the file cap` explaining specifically why splitting would be worse.
- The diff is mechanical (rename, codemod, bulk format) — not 30 files of new logic.

If the PR is over the cap because it's "just a lot of small features bundled together," it's not atomic — split it.

## Common rationalizations

| Excuse | Reality |
|---|---|
| "All these files are one feature, splitting feels arbitrary" | The reviewer doesn't care that it's "one feature." They care about reading time. Split. |
| "Part 1 won't compile / tests won't pass without Part 2" | Then your seam is wrong. Re-plan the split so each part stands alone. |
| "It's only 22 files, that's basically 20" | 20 is the cap, not 22. Split. |
| "I already opened the PR" | Close it. Open Part 1 and Part 2. Reviewer time saved > your re-pushing cost. |
| "The plan author didn't split it" | Then the plan was wrong. Update the plan, then split. |
| "I'll just add a feature flag and ship it all" | A flag dance for a non-atomic change is more code than the split. Split. |

## Red flags — STOP

- You're about to push 25+ files in one PR.
- You're rationalizing why this one is special.
- You're considering hiding the size by squashing or by not pushing tests.

All of these mean: **stop, re-plan the split, open Part 1 only.**
